# Code Of Conduct

CloudX aims to be a technically rigorous, public-facing, and welcoming open-source project.

Participation in this repo should remain respectful, constructive, and safe for contributors with different backgrounds, identities, skill levels, and viewpoints.

## Our Standards

Examples of behavior that help this project:

- being respectful in disagreement
- giving specific, actionable technical feedback
- assuming good intent while still being honest about problems
- accepting correction gracefully
- explaining tradeoffs instead of escalating conflict
- protecting the privacy and safety of other contributors

Examples of unacceptable behavior:

- harassment, intimidation, or personal attacks
- discriminatory, exclusionary, or degrading language
- sexualized language or unwelcome attention
- publishing someone else’s private information without permission
- repeated bad-faith disruption, trolling, or hostility
- using technical disagreement as cover for abusive behavior

## Scope

This code of conduct applies to:

- GitHub issues
- pull requests
- code review comments
- project discussions
- public spaces where someone is clearly representing CloudX

## Enforcement

Maintainers may edit, hide, lock, or remove comments or contributions that violate this policy. They may also reject contributions or remove participants from project spaces when behavior repeatedly or seriously violates these standards.

General conduct concerns should be reported privately to the project maintainer through a non-public channel. Security concerns should follow the reporting guidance in [SECURITY.md](SECURITY.md) instead of being raised as ordinary public discussion.

## Working Norms

CloudX favors:

- evidence over volume
- directness over performative politeness
- clear ownership over ambiguity
- technical critique over personal critique
- correction over defensiveness

Contributors do not need to agree on every implementation choice, but they do need to work in a way that remains safe and productive for everyone else.

## Enforcement Ladder

The typical progression is:

1. correction
2. warning
3. temporary restriction or removal
4. permanent removal from project spaces

Maintainers may skip steps when behavior is severe enough to warrant stronger action immediately.

## Attribution

This document is informed by the Contributor Covenant spirit, but written to match the actual working style of the CloudX project: technically direct, collaborative, and public-repo safe.
